## 1. TL;DR и общий вердикт

У проэкта **добротный MVP с грамотно спроектированной архитектурой**: E2EE по Signal-протоколу (X3DH + Double Ratchet) реализован корректно, параметризованные SQL-запросы (нет SQL-инъекций), helmet + express-rate-limit подключены, bcrypt cost = 12, httpOnly cookie выставляется, constant-time сравнения в крипто, WebSocket authentication через cookie. Crypto-тесты качественные и реально тестируют libsodium без моков.

**Но открывать публичный доступ в текущем виде нельзя.** Найдено:
- **5 критических уязвимостей безопасности** (JWT в localStorage, нет MIME-валидации upload, IDOR в download, WS-signaling без проверки chat membership, reply_to без проверки чата)
- **4 блокера** в error handling, ведущих к молча пропадающим сообщениям (broadcastToChat без await, chatMemberCheck async без try/catch, decryptReceived глотает MITM, Ed25519 signing key не сохраняется)
- **43 места** глотания ошибок (`catch {}`, `.catch(() => {})`, пустые fallback'и)
- **6 SQL/DB проблем** (индекс messages создан в неправильном порядке, ILIKE поиск seq scan, N+1 в /api/chats, нет миграций, N+1 в sendPushToOfflineMembers)
- **5 инфра-проблем** (MinIO URL захардкожен на localhost, нет Sentry/observability, нет offsite backups, graceful shutdown не ждёт активных запросов, Redis без persistence)
- **Тесты покрывают только 43% API endpoints**, rate limiting не тестируется, WebSocket/WebRTC signaling без тестов, E2E Playwright не в CI

**Зрелость подсистем (0–5):**

## 3. CRITICAL — блокеры продакшна

Эти проблемы **должны быть исправлены до открытия публичного доступа**. Phase 0 hotfix, 6–7 дней.

### 3.1. Security критичные

#### SEC-1. JWT в localStorage (XSS → угон аккаунта на 7 дней)

**Файлы:**
- `webApp/src/api/client.ts:10` — `getToken()` возвращает `memoryToken || localStorage.getItem('son-token')`
- `webApp/src/stores/authStore.ts:31` — `login()` делает `localStorage.setItem('son-token', token)`

**Суть:** комментарий в `client.ts:5` обещает "JWT хранится в памяти, НЕ в localStorage (защита от XSS)" — но реальность прямо противоположна. Любой XSS (уязвимая зависимость, CDN компромис) получает токен через `localStorage.getItem('son-token')` и использует его 7 дней.

**Примечание:** это коммит `da01148 fix(web): читать токен из localStorage как fallback для WebView` — был сделан ради Tauri/WebView совместимости. Это **регрессия безопасности**, нужно откатить и решить проблему WebView иначе.

**Как исправить:**
```ts
// webApp/src/api/client.ts
export function getToken(): string | null {
  return memoryToken;  // только память
}

// webApp/src/stores/authStore.ts
login: (token, user) => {
  setToken(token);
  // localStorage.setItem('son-token', token) — УБРАТЬ
  localStorage.setItem('son-user', JSON.stringify(user));
  set({ token, user, isAuthenticated: true });
},

restore: () => {
  // Не читать токен из localStorage.
  // Вместо этого — GET /api/users/me с credentials:'include',
  // если 200 → user в cookie, ok.
  // Если 401 → logout.
}
```

Для Tauri/WebView: передавать токен через Tauri command API или использовать Tauri storage plugin с native keychain, а не localStorage.

#### SEC-2. Нет валидации MIME/magic bytes при upload + path injection в MinIO

**Файлы:**
- `server/index.js:232-242` — `POST /api/users/me/avatar`
- `server/index.js:1325-1343` — `POST /api/media/upload`
- `server/index.js:1361-1363` — `POST /api/media/upload-url`
- `server/storage.js:36-49` — `uploadFile`

**Суть:**
- `req.file.mimetype` берётся из заголовка клиента без проверки реального содержимого
- `folder` берётся напрямую из `req.body.folder` без whitelist → path injection (`folder=../../admin`)
- `ext` берётся из имени файла → можно сохранить `file.php`

**Как исправить:**
```js
const ALLOWED_FOLDERS = ['attachments', 'images', 'voice', 'avatars'];
const ALLOWED_MIME = new Set([
  'image/jpeg', 'image/png', 'image/gif', 'image/webp',
  'audio/webm', 'audio/ogg', 'audio/mp4', 'audio/mpeg',
  'video/mp4', 'video/webm',
  'application/pdf',
]);

// В upload-handler'е:
const folder = ALLOWED_FOLDERS.includes(req.body.folder) ? req.body.folder : 'attachments';

if (!ALLOWED_MIME.has(req.file.mimetype)) {
  return res.status(400).json({ error: 'Недопустимый тип файла' });
}

// Magic bytes:
const { fileTypeFromBuffer } = await import('file-type');
const detected = await fileTypeFromBuffer(req.file.buffer);
if (!detected || !ALLOWED_MIME.has(detected.mime)) {
  return res.status(400).json({ error: 'Тип файла не соответствует содержимому' });
}
```

Добавить зависимость `file-type` в `server/package.json`.

#### SEC-3. IDOR в `POST /api/media/download` — чужие файлы доступны

**Файл:** `server/index.js:1347-1355`

**Суть:**
```js
app.post('/api/media/download', authMiddleware, async (req, res) => {
  const objectName = req.body.object_name;
  const url = await getDownloadUrl(objectName);  // нет проверки ownership!
  res.json({ url });
});
```

Любой аутентифицированный пользователь может запросить presigned URL для объекта другого пользователя, подставив его `object_name` в body.

**Как исправить:**
```js
app.post('/api/media/download', authMiddleware, async (req, res) => {
  const objectName = req.body.object_name;
  if (!objectName || typeof objectName !== 'string' ||
      objectName.includes('..') || objectName.startsWith('/')) {
    return res.status(400).json({ error: 'Недопустимое имя объекта' });
  }
  // Проверка: либо attachments, либо avatar принадлежит пользователю
  const r = await pool.query(`
    SELECT 1 FROM attachments a
    JOIN messages m ON m.id = a.message_id
    JOIN chat_members cm ON cm.chat_id = m.chat_id
    WHERE a.object_name = $1 AND cm.user_id = $2
    LIMIT 1
  `, [objectName, req.user.id]);
  if (r.rows.length === 0) {
    return res.status(403).json({ error: 'Нет доступа к файлу' });
  }
  const url = await getDownloadUrl(objectName);
  res.json({ url });
});
```

#### SEC-4. WebSocket signaling без проверки общего чата

**Файл:** `server/index.js:1109-1158`

**Суть:** `call_offer`, `call_answer`, `ice_candidate`, `call_end`, `call_reject` вызывают `sendToUser(msg.target_user_id, ...)` без проверки, что `target_user_id` входит в общий чат с отправителем. Любой аутентифицированный пользователь может звонить/слать SDP/ICE кому угодно по user_id.

**Как исправить:**
```js
// Утилита для проверки общего чата
async function haveSharedChat(userA, userB) {
  const r = await pool.query(`
    SELECT 1 FROM chat_members cm1
    JOIN chat_members cm2 ON cm1.chat_id = cm2.chat_id
    WHERE cm1.user_id = $1 AND cm2.user_id = $2
    LIMIT 1
  `, [userA, userB]);
  return r.rows.length > 0;
}

// В обработчике WS signaling:
case 'call_offer':
case 'call_answer':
case 'ice_candidate':
case 'call_end':
case 'call_reject': {
  if (!await haveSharedChat(user.id, msg.target_user_id)) return;
  sendToUser(msg.target_user_id, { ...msg, from_user_id: user.id });
  break;
}
```

#### SEC-5. `reply_to` не проверяется на принадлежность чату

**Файл:** `server/index.js:623, 689-690`

**Суть:** при отправке сообщения `reply_to` принимается из body и вставляется в БД без проверки, что referenced сообщение принадлежит тому же чату. Пользователь из чата A может указать `reply_to` с ID сообщения из чата B (в т.ч. секретного), раскрывая ID и нарушая изоляцию.

**Как исправить:**
```js
if (reply_to) {
  const r = await pool.query(
    'SELECT id FROM messages WHERE id = $1 AND chat_id = $2',
    [reply_to, chatId]
  );
  if (r.rows.length === 0) {
    return res.status(400).json({ error: 'Сообщение для ответа не найдено в этом чате' });
  }
}
```

### 3.2. Надёжность доставки сообщений

#### ERR-1. `broadcastToChat` и `sendPushToOfflineMembers` fire-and-forget

**Файл:** `server/index.js:707-709`

```js
await client.query('COMMIT');
broadcastToChat(chatId, { type: 'new_message', message: msg }, req.user.id);
sendPushToOfflineMembers(req.params.chatId, req.user.id, content, req.user.display_name);
res.status(201).json(msg);
```

Обе функции — async с SQL-запросами внутри. При ошибке БД rejection улетает в `unhandledRejection`. Отправитель получает `201 OK`, но **остальные участники не получат сообщение через WebSocket, офлайн — не получат push**. Классическая проблема "исчезающих сообщений".

**Как исправить:**
```js
try {
  await broadcastToChat(chatId, { type: 'new_message', message: msg }, req.user.id);
} catch (err) {
  logger.error({ err, chatId, msgId: msg.id }, 'broadcastToChat failed');
  // Sentry.captureException(err)
}
sendPushToOfflineMembers(req.params.chatId, req.user.id, content, req.user.display_name)
  .catch(err => logger.error({ err, chatId }, 'push send failed'));
```

Плюс установить глобальный `process.on('unhandledRejection', ...)`.

#### ERR-2. `chatMemberCheck` async middleware без try/catch

**Файл:** `server/index.js:443-454`

```js
async function chatMemberCheck(req, res, next) {
  const chatId = req.params.chatId;
  const userId = req.user.id;
  const result = await pool.query(
    'SELECT 1 FROM chat_members WHERE chat_id = $1 AND user_id = $2',
    [chatId, userId]
  );
  if (result.rows.length === 0) return res.status(403).json({ error: 'Нет доступа к этому чату' });
  next();
}
```

Express 4/5 не ловит rejection из async middleware. При ошибке БД запрос виснет до client timeout, rejection улетает в `unhandledRejection`. Под нагрузкой может уронить процесс.

**Как исправить:**
```js
// В начале server/index.js:
require('express-async-errors');

async function chatMemberCheck(req, res, next) {
  try {
    const r = await pool.query(/* ... */);
    if (r.rows.length === 0) {
      return res.status(403).json({ error: 'Нет доступа к этому чату' });
    }
    next();
  } catch (err) {
    logger.error({ err, userId: req.user.id, chatId: req.params.chatId }, 'chatMemberCheck DB error');
    res.status(503).json({ error: 'Сервис временно недоступен' });
  }
}
```

#### ERR-3. `decryptReceived` глотает ошибку дешифровки (возможный MITM невидим)

**Файл:** `webApp/src/stores/secretChatStore.ts:216-242`


#### ERR-4. Ed25519 signing key не сохраняется в IndexedDB

**Файл:** `webApp/src/stores/secretChatStore.ts:73-111`

При `initialize()` генерируется `mySigningKey` (Ed25519), но сохраняется только `myIdentityKey` (X25519) через `saveKeyPair('identity', ...)`. После F5 `restoreSessions` восстанавливает identity, но signing — `null`. При следующем `initialize()` генерируются новые ключи → старые сессии становятся недоступны, на сервере публикуется новый identity без уведомления собеседников.

**Как исправить:**
```ts
// В initialize():
await saveKeyPair('identity', identityKey);
await saveKeyPair('signing', signingKey);  // добавить

// В restoreSessions():
const signingKey = await loadKeyPair('signing');
if (!identityKey || !signingKey) {
  // Требуется повторная инициализация
  return;
}
set({ myIdentityKey: identityKey, mySigningKey: signingKey, initialized: true });
```

#### ERR-5. Plaintext содержимое в push для секретных чатов

**Файл:** `server/index.js:707-709`

Для секретного чата `dbContent = payload.ciphertext` (на строке 665), но в `sendPushToOfflineMembers` передаётся оригинальный `content` из body, который может содержать plaintext или JSON с ciphertext. В любом случае — неподходящее для push уведомления.

**Как исправить:**
```js
const isSecret = chat.type === 'secret';
const pushContent = isSecret ? null : content;
sendPushToOfflineMembers(chatId, req.user.id, pushContent, req.user.display_name)
  .catch(err => logger.error({ err, chatId }, 'push failed'));
```

В `sendPushToOfflineMembers` — при `pushContent === null` использовать `"🔒 Зашифрованное сообщение"`.

### 3.3. База данных

#### DB-1. Индекс `idx_messages_chat` создан в ASC-порядке, все запросы DESC

**Файл:** `server/db.js:155` (примерно)

```sql
CREATE INDEX IF NOT EXISTS idx_messages_chat ON messages(chat_id, created_at);
```

Все запросы к сообщениям используют `ORDER BY m.created_at DESC`. PostgreSQL умеет backward scan, но стоимостно хуже. Критично для hot path.

**Как исправить:** см. раздел [8.1](#81-sql-скрипт-для-phase-0-применить-на-живой-бд) SQL-скрипт.

#### DB-2. Поиск сообщений через `ILIKE '%q%'` — seq scan

**Файл:** `server/index.js:1387-1397`

```sql
WHERE m.content ILIKE $2  -- $2 = '%текст%'
```

С ведущим `%` никакой B-tree индекс не работает. При 1M сообщений — seq scan всей таблицы messages на каждый поисковый запрос. Rate-limit 200 req/min на весь API → поиск можно дёргать интенсивно, DoS-возможность.

**Как исправить:** добавить FTS колонку + GIN индекс (см. SQL-скрипт), поменять запрос на `WHERE m.content_tsv @@ plainto_tsquery('russian', $1)`.

#### DB-3. N+1 в `GET /api/chats` — коррелированные подзапросы

**Файл:** `server/index.js:504-521`

Для каждого чата пользователя выполняются 2 коррелированных подзапроса (`json_agg` members + `json_build_object` last_message). 50 чатов × 2 = 100 доп. lookup на запрос.

**Как исправить:** `LEFT JOIN LATERAL` (см. раздел [8.2](#82-рефакторинг-ключевых-sql-запросов)).

#### DB-4. N+1 в `sendPushToOfflineMembers`

**Файл:** `server/index.js:1188-1215`

```js
for (const member of members.rows) {
  const tokens = await pool.query('SELECT ... FROM push_tokens WHERE user_id = $1', [member.user_id]);
  // ...
}
```

При 500 участников в группе — 500 SELECT подряд, блокируя event loop.

**Как исправить:** один JOIN-запрос.

```js
const result = await pool.query(`
  SELECT pt.endpoint, pt.p256dh, pt.auth
  FROM chat_members cm
  JOIN push_tokens pt ON pt.user_id = cm.user_id
  WHERE cm.chat_id = $1 AND cm.user_id != $2
`, [chatId, senderId]);
```

#### DB-5. `broadcastToChat` — DB-запрос на каждое WS-событие

**Файл:** `server/index.js:1173-1185`

Каждый `typing`, `read`, `new_message` делает SELECT участников. В активном чате — десятки SQL/сек.

**Как исправить:** кешировать `chat_id → [user_ids]` в памяти (Map с TTL ~30 сек) или Redis. При горизонтальном scale — обязательно Redis Pub/Sub.

#### DB-6. Нет системы миграций

**Файл:** `server/db.js`

Схема инициализируется через `CREATE TABLE IF NOT EXISTS` + `DO $$ BEGIN ALTER TABLE ... EXCEPTION WHEN OTHERS THEN NULL END $$`. Нет истории изменений, нет rollback, нет атомарного применения.

**Как исправить:** установить `node-pg-migrate`, вынести текущую схему в `server/migrations/001_initial.sql`, убрать DDL-код из `db.js`. В `start()` вызывать `runMigrations()`.

### 3.4. Инфраструктура

#### INF-1. MinIO URL захардкожен на `http://localhost:9000`

**Файл:** `server/storage.js:46` (примерно)

```js
url: `http://localhost:9000/${BUCKET}/${objectName}`
```

При работе через Cloudflare Tunnel клиенты не могут достучаться до `localhost:9000` VPS. Все аватары и presigned URL для медиа **ломаются** на проде (если пользователь видит аватар, значит публичная политика bucket + прокси через nginx — уточнить).

**Как исправить:**
```js
const MINIO_PUBLIC_URL = process.env.MINIO_PUBLIC_URL || `http://localhost:9000`;
// ...
url: `${MINIO_PUBLIC_URL}/${BUCKET}/${objectName}`
```

В prod `.env` установить `MINIO_PUBLIC_URL=https://media.sonchat.uk` + настроить Cloudflare Tunnel route на MinIO (либо проксировать через nginx в web-контейнере).

#### INF-2. Нет Sentry / observability

Только `console.error/log` в stdout. Нет request-id, нет стектрейсов в агрегаторе, нет метрик.

**Как исправить:** см. Phase 1.

#### INF-3. Graceful shutdown не awaited

**Файл:** `server/index.js:1492` (примерно)

```js
server.close();  // не awaited
pool.end();      // не awaited
process.exit(0);
```

При SIGTERM активные HTTP-запросы могут получить 503 / разорванное соединение, пропавшие сообщения.

**Как исправить:**
```js
async function shutdown(signal) {
  logger.info({ signal }, 'shutdown started');
  await new Promise(resolve => server.close(resolve));
  wss.clients.forEach(ws => ws.close(1001, 'Server shutdown'));
  await pool.end();
  await redisClient?.quit();
  logger.info('shutdown complete');
  process.exit(0);
}
process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```

#### INF-4. Health check проверяет только postgres

**Файл:** `server/index.js:1463-1473`

`/health` проверяет только pg.query, но не Redis и не MinIO. Если MinIO упадёт — `/health` = 200, а upload файлов падает.

**Как исправить:** разделить на `/health/live` (процесс жив) и `/health/ready` (все зависимости доступны).

#### INF-5. Redis без persistence

**Файл:** `docker-compose.prod.yml:41`

```yaml
command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
```

Нет `--appendonly yes`. При рестарте — все кэши/rate-limit сбрасываются.

**Как исправить:** `--appendonly yes --appendfsync everysec`.

### 8.1. SQL-скрипт для Phase 0 (применить на живой БД)

```sql
-- ============================================================
-- sOn Messenger — Production DB Hardening (Phase 0)
-- Применять с опцией CONCURRENTLY, безопасно на работающей БД
-- ============================================================

-- 1) Исправить направление индекса для горячего пути загрузки сообщений
DROP INDEX CONCURRENTLY IF EXISTS idx_messages_chat;
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_messages_chat_id_created
    ON messages(chat_id, created_at DESC);

-- 2) Partial index для one_time_prekeys — ускорение выдачи OPK
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_otpk_unused
    ON one_time_prekeys(user_id, created_at ASC)
    WHERE used = false;

-- 3) Индекс на sender_id — для JOIN с users в некоторых запросах
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_messages_sender
    ON messages(sender_id);

-- 4) Индекс на call_history(created_at DESC) — сортировка истории звонков
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_call_history_created
    ON call_history(created_at DESC);

-- 5) Full-Text Search для messages.content (замена ILIKE)
ALTER TABLE messages
    ADD COLUMN IF NOT EXISTS content_tsv tsvector
    GENERATED ALWAYS AS (to_tsvector('russian', coalesce(content, ''))) STORED;

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_messages_fts
    ON messages USING GIN(content_tsv);

-- После применения заменить в server/index.js:
--   WHERE m.content ILIKE $2      -- старое
--   WHERE m.content_tsv @@ plainto_tsquery('russian', $2)  -- новое

-- 6) FK для messages.reply_to
ALTER TABLE messages
    DROP CONSTRAINT IF EXISTS fk_messages_reply_to,
    ADD CONSTRAINT fk_messages_reply_to
        FOREIGN KEY (reply_to) REFERENCES messages(id) ON DELETE SET NULL;

-- 7) CHECK constraints для enum-подобных VARCHAR полей
ALTER TABLE chats
    DROP CONSTRAINT IF EXISTS chats_type_check,
    ADD CONSTRAINT chats_type_check
        CHECK (type IN ('direct', 'group', 'secret'));

ALTER TABLE messages
    DROP CONSTRAINT IF EXISTS messages_type_check,
    ADD CONSTRAINT messages_type_check
        CHECK (type IN ('text', 'image', 'file', 'audio', 'video', 'voice', 'system'));

ALTER TABLE messages
    DROP CONSTRAINT IF EXISTS messages_status_check,
    ADD CONSTRAINT messages_status_check
        CHECK (status IN ('sent', 'delivered', 'read', 'failed'));

-- 8) CHECK на длину content (защита на уровне БД)
ALTER TABLE messages
    DROP CONSTRAINT IF EXISTS messages_content_length,
```

### 8.2. Рефакторинг ключевых SQL-запросов

#### 8.2.1. `sendPushToOfflineMembers` — устранить N+1

**Было** (`server/index.js:1188-1215`, концептуально):
```js
for (const member of members.rows) {
  const tokens = await pool.query('SELECT ... FROM push_tokens WHERE user_id = $1', [member.user_id]);
  // ...
}
```

**Стало:**
```js
async function sendPushToOfflineMembers(chatId, senderId, content, senderName) {
  const result = await pool.query(`
    SELECT pt.endpoint, pt.p256dh, pt.auth, pt.user_id
    FROM chat_members cm
    JOIN push_tokens pt ON pt.user_id = cm.user_id
    LEFT JOIN users u ON u.id = cm.user_id
    WHERE cm.chat_id = $1
      AND cm.user_id != $2
      AND (u.is_online = false OR u.last_seen_at < NOW() - INTERVAL '2 minutes')
  `, [chatId, senderId]);

  const results = await Promise.allSettled(
    result.rows.map(row =>
      webpush.sendNotification(
        { endpoint: row.endpoint, keys: { p256dh: row.p256dh, auth: row.auth } },
        JSON.stringify({ title: senderName, body: content ?? '🔒 Зашифрованное сообщение' })
      )
    )
  );

  // Удалить expired tokens одним запросом
  const expiredEndpoints = [];
  results.forEach((r, i) => {
    if (r.status === 'rejected' && r.reason?.statusCode === 410) {
      expiredEndpoints.push(result.rows[i].endpoint);
    }
  });
  if (expiredEndpoints.length > 0) {
    await pool.query('DELETE FROM push_tokens WHERE endpoint = ANY($1::text[])', [expiredEndpoints]);
  }
}
```

#### 8.2.2. `GET /api/chats` — LATERAL joins вместо коррелированных подзапросов

```sql
SELECT c.id, c.type, c.name, c.description, c.avatar_url,
       c.last_message_at, c.created_at,
       cm.unread_count,
       members_agg.members,
       lm.last_message
FROM chats c
JOIN chat_members cm ON cm.chat_id = c.id AND cm.user_id = $1
LEFT JOIN LATERAL (
    SELECT json_agg(json_build_object(
        'id', u.id,
        'display_name', u.display_name,
        'avatar_url', u.avatar_url,
        'is_online', u.is_online
    )) AS members
    FROM chat_members cm2
    JOIN users u ON u.id = cm2.user_id
    WHERE cm2.chat_id = c.id
) members_agg ON true
LEFT JOIN LATERAL (
    SELECT json_build_object(
        'content', CASE WHEN c.type = 'secret'
                        THEN '🔒 Зашифрованное сообщение'
                        ELSE m.content END,
        'created_at', m.created_at,
        'sender_id', m.sender_id,
        'type', m.type
    ) AS last_message
    FROM messages m
    WHERE m.chat_id = c.id
    ORDER BY m.created_at DESC
    LIMIT 1
) lm ON true
ORDER BY c.last_message_at DESC NULLS LAST, c.created_at DESC;
```

С индексом `idx_messages_chat_id_created(chat_id, created_at DESC)` LATERAL использует Index Scan + LIMIT 1 для каждого чата — O(log N) вместо O(N).
